**Architecture decision (confirmed Week 1)**:

R1 follows ABot-Claw's proven pattern. The LLM reads SDK docs (Adapter methods + Week 1 data contracts) and generates Python code directly calling `env.stand()` / `env.move_velocity()` / `env.read_cameras()`. Safety is provided by lease TTL, CodeValidator, Critic feedback, and SDK2-level safety. No intermediate JSON command translation layer.

Consequences for Week 2:
- The `R1RobotAdapter` Protocol (`interfaces/robot_adapter.py`) is the single typed execution seam. Its methods mirror `PiperRobotEnv`: `stand()`, `move_velocity()`, `stop()`, `damp()`, `read_cameras()`, `get_state()`, `probe_capabilities()`.
- Week 1 contracts (Telemetry, RobotState, CapabilityReport, CameraFrameMetadata, AudioChunkMetadata) serve as the **data-definition side of R1's SDK docs**. The LLM reads them alongside the Adapter methods to know what `env.get_state()` returns and what `env.read_cameras()` produces.
- `contracts/skill-command.schema.json` is kept as reference documentation of valid parameter ranges — not as an execution interface.
